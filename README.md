# Cs230

Summary of The Gaming Room and Their Requirements

The Gaming Room is a client that wanted to bring their existing game, Draw It or Lose It, to a web based, multi platform environment. Their main requirement was to design a system that supports multiple teams and players, ensures unique game and team names, and maintains a single active instance of the game in memory. They also needed a scalable architecture that could support distributed access across different operating platforms.


What I Did Well

I communicated the design clearly and organized the document in a way that made it easy for both technical and non technical stakeholders to understand. I also explained the design constraints and architectural decisions in a way that connected directly to the client’s needs, which helped justify the choices made in the design.


What Helped When Developing the Code

Working through the design document first made the coding process much easier. Defining the classes, relationships, and constraints ahead of time gave me a clear roadmap. The UML diagram and the focus on design patterns, especially the Singleton pattern, helped ensure the code stayed consistent with the client’s requirements.


What I Would Revise

If I were to revise one part, I would refine the evaluation section by adding more detail about trade offs between different operating platforms. I would expand on performance, security, and scalability considerations to make the comparison even stronger.


Interpreting and Implementing User Needs

I focused on the client’s core needs: uniqueness of game and team names, a single game instance, and support for multiple users. These needs shaped the structure of the classes and the use of the Singleton pattern. Considering user needs is essential because it ensures the final product solves the right problems and provides a smooth experience for both the client and the end users.


My Approach to Software Design

I approached the design by breaking the system into clear components, identifying constraints, and using established design patterns. I relied on UML modeling, iterative refinement, and alignment with the client’s requirements. In the future, I would continue using these strategies and also incorporate more early prototyping and user feedback to validate design decisions sooner.
